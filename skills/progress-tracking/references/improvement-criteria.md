# Improvement Criteria

## 什么算"实质性改进"

### 判断标准

实质性改进必须满足以下至少一条：

1. **核心功能修复**：上次评价中指出的功能缺陷被修复，且有对应的测试验证。
2. **设计问题解决**：架构或设计层面的问题被重构，而非简单打补丁。
3. **关键 bug 修复**：影响正确性的 bug 被定位并修复，附带 root cause 分析。
4. **测试覆盖提升**：针对上次指出的未覆盖场景，新增了有意义的测试用例。
5. **性能/可靠性改进**：有 benchmark 数据支撑的性能优化或容错机制改进。

### 量化参考（非硬性门槛）

| 维度 | 最低期望 |
|------|---------|
| 解决的关键问题数 | >= 1 |
| 新增有效代码行数 | >= 20 行（排除空行和注释） |
| 新增测试用例 | >= 1 个针对性测试 |
| Commit 数量 | >= 2 个有意义的 commit（非 typo fix） |

## 常见的"虚假改进"模式

### 1. 只改 README / 文档
- 表现：大量 commit 但只修改了 README.md、添加了注释
- 判断：除非上次的核心问题就是"缺少文档"，否则不算实质性改进
- 标记：`improved=false`，备注"文档改善但核心问题未解决"

### 2. 只加注释不改逻辑
- 表现：在代码中添加大量中文注释，但算法逻辑、bug 未修改
- 判断：注释有助于可读性，但不解决功能性问题
- 标记：`improved=false`，备注"可读性改善但功能问题未修复"

### 3. 复制代码但不理解
- 表现：从 Stack Overflow / ChatGPT 复制了代码块，但集成方式不当
- 信号：新增代码与项目风格不一致、未处理边界情况、无对应测试
- 判断：需要验证新代码是否真正解决了问题
- 标记：视实际效果判断

### 4. 重复提交相同内容
- 表现：多次提交但 diff 内容几乎相同，或 revert 后再 commit
- 判断：无实质进展
- 标记：`improved=false`，备注"提交历史显示无新增改动"

### 5. 格式化刷 commit
- 表现：运行 formatter/linter 产生大量 diff，但业务逻辑未变
- 信号：大量文件被修改但改动内容为空格、缩进、import 排序
- 判断：代码质量微改善，但不解决核心问题
- 标记：`improved=false`，备注"格式优化但核心问题未解决"

## 改进方向优先级

当给出下一步行动建议时，按以下优先级排列：

1. **功能正确性（Functionality）**：核心功能是否正确实现、bug 是否修复
2. **测试覆盖（Testing）**：关键路径是否有测试、edge case 是否覆盖
3. **文档完整性（Documentation）**：API 文档、使用说明、设计文档
4. **代码质量（Code Quality）**：命名、结构、可维护性
5. **格式规范（Formatting）**：代码风格、linting 规则

> 原则：优先解决"程序能不能正确工作"，再关注"代码好不好看"。
